系统的安全哲学

这套系统有一条贯穿始终的原则：自动化是"变化提醒"，不是"攻击武器"。

✅
默认只做被动收集

subfinder、dnsx、httpx、低风险 Nuclei——这些都是"看"，不是"打"

🚫
默认排除危险操作

DoS、brute-force、intrusive、RCE、LFI、SQLi 相关的 Nuclei 模板全部排除

🔐
必须先确认授权

scope 配置里 legal_ack=true 才能运行 pipeline，这是硬性门槛

👁️
人工验证是必须的最后一步

所有自动化输出只是"候选"，报告前必须手工确认真实影响

⚠️
为什么这很重要？

Bug Bounty 是在授权范围内测试。越界测试不只是违规——它可能触犯法律。这套系统的每个默认值都在保护你，不要绕过它们。

Token 是什么？

Token 是 AI 模型处理文本的基本单位，也是费用的计量单位。这个提示词被精心设计，让 Grok 跳过无用内容，节省你的 API 费用。

一张技巧卡片有什么

情报经过 Grok 提炼后，变成标准化的"技巧卡片"存进知识库。每张卡都有固定的字段：

标题 简短的技巧名称，如"invoice_id IDOR via export endpoint"
漏洞类型 IDOR、OAuth bypass、GraphQL mass assignment 等分类
前置条件 需要两个账号？需要付费订阅？需要特定 API 版本？
核心 trick 具体操作步骤，可以直接用于测试
可迁移目标 这个 trick 最可能在哪类平台上奏效
来源 URL 每条声明必须有来源——这是系统的质量底线
置信度 / 风险等级 high/medium/low；low/medium/high——帮你决定优先测哪个

关键工程决策

知识库里的每条记录都有来源 URL。grok_kb_validate.py 会拒绝任何没有来源的卡片。这防止了 AI 幻觉：Grok 生成内容，必须能指向真实的来源才算数。

自动化侦察引擎

用 VPS 每天静默扫描——只在"有变化"时才叫醒你

它是一个永不睡觉的助手

想象你雇了一个助手，他的工作是：每天凌晨3点去目标公司门口，记录门口有几辆车、多少员工进出、有没有新开的侧门——然后只有当今天和昨天不一样时，才来敲你的门。

这就是 recon_pipeline.py 的工作模式。它在你睡觉的时候运行，发现变化时才通过 Discord 或 Telegram 通知你。

🖥️
什么是 VPS？

VPS 是你在互联网上租的一台永远开着的服务器，可以7×24小时运行脚本。

Pipeline 的7个步骤

每次运行 recon_pipeline.py，按这个顺序执行：

1
检查授权（legal_ack 守门）

读取 scope 配置，legal_ack=false 时直接退出，拒绝运行。这是第一道护栏。

2
随机延迟启动

随机等待1-4秒再开始，避免每次都在整点触发，减少被目标检测到的风险

3
被动子域名收集（subfinder）

subfinder 从公开数据源（证书透明度日志、DNS 数据库）找出所有子域名，完全被动

4
Scope 过滤（scope_guard）

每个发现的子域名都过 scope_guard.py，不在白名单里的全部丢弃

5
DNS 解析（dnsx）

DNS 解析确认哪些子域名实际上有服务器在响应，过滤掉已失效的域名

6
HTTP 探活（httpx）

httpx 探测哪些资产有 Web 服务，收集页面标题、技术栈信息，速率限制25次/秒

7
低风险 Nuclei 扫描

只跑 exposure、misconfig、takeover、tech 标签，排除所有 dos/rce/sqli/exploit 相关模板

scope 配置文件如何控制一切

整个 pipeline 的行为由一个 JSON 配置文件控制。这是真实的 scope.example.json 里的关键部分：

CODE 
{
  "legal_ack": false,
  "in_scope_domains": [
    "example.com",
    "*.example.com"
  ],
  "out_of_scope_domains": [
    "*.internal.example.com"
  ],
  "recon": {
    "active_crawl": false,
    "nuclei_low_risk": true
  },
  "limits": {
    "httpx_rate": 25
  }
}
PLAIN ENGLISH

开始这个大括号里的配置

false = 不运行！只有你确认授权后才改成 true

允许测试的域名列表开始

example.com 本身可以测试

所有 *.example.com 子域名（如 api.example.com）都可以

绝对不能碰的域名列表

internal 子域名是内部系统，不能碰

侦察行为设置

主动爬虫关闭——只被动收集，不主动爬取页面

低风险 Nuclei 扫描开启

速率限制设置

httpx 每秒最多发25个请求，不轰炸目标服务器

scope_guard.py 如何过滤域名

这是 scope_guard.py 里最关键的逻辑——只有30行，但它是整个系统的安全底线：

CODE 
def is_allowed(host: str, cfg: dict) -> bool:
    ins = cfg.get('in_scope_domains', [])
    outs = cfg.get('out_of_scope_domains', [])
    if not any(host_matches(host, p) for p in ins):
        return False
    if any(host_matches(host, p) for p in outs):
        return False
    return True
PLAIN ENGLISH

定义一个函数：给定域名和配置，判断能不能测

读取配置里的白名单域名列表

读取配置里的黑名单（排除）域名列表

如果这个域名不在白名单里的任何一条记录里...

...立刻返回"不允许"（宁可漏掉，也不越界）

如果这个域名匹配黑名单里的任何一条记录...

...也返回"不允许"（明确禁止就是禁止）

两道检查都过了，才返回"允许"

💡
"双重否定"设计

注意这个函数先检查"是否在白名单"，再检查"是否在黑名单"。这是"默认拒绝"原则——不明确允许的域名，一律不测。这比"只检查黑名单"安全得多，因为你不可能事先列出所有不该测的域名。

培养调试直觉

系统不能用了。你不知道哪里出了问题。大多数人这时候会感到恐慌——但这套系统被设计得可以系统性地诊断问题。

就像医生诊断病人：不是随机猜测，而是按症状缩小范围。先问"哪一层出了问题？"

1
配置层问题

pipeline 直接报错退出 → 检查 legal_ack 是否为 true、scope JSON 格式是否正确

2
工具层问题

某步骤没有输出 → 检查对应工具是否安装（which subfinder、which httpx）

3
Scope 过滤太严

subdomains.scoped.txt 为空 → 用 make check-scope 验证 scope 配置能否匹配你的种子域名

4
API 层问题（Grok Agent）

情报收集失败 → 检查 .env 里的 API key 是否有效、是否超出速率限制

最有价值的调试工具：dry-run

系统提供了一个 --dry-run 模式，它会打印出"如果真的运行会执行什么命令"，但不实际执行。就像飞行模拟器——帮你练习而不冒真实风险。

CODE 
# 安全测试：不会真的跑任何工具
python3 scripts/recon_pipeline.py \
  --config config/scope.example.json \
  --dry-run

# 验证 scope 是否配置正确
make check-scope

# 查看项目目录结构
make tree

Makefile 是什么？

Makefile 是命令的快捷方式集合。当你不记得某个脚本的完整参数时，make tree 比find . -maxdepth 3 -type f | sort容易记多了。

提交报告前的安全检查

在你按下"提交"之前，这套系统有一张最终检查清单。每一条都是血泪教训的总结：

✅ 授权确认 这个资产在项目的官方 scope 里，不只是"看起来像"
✅ 双账号验证 用你自己的两个测试账号复现，不是用真实用户数据
✅ 影响证明 能说清楚"攻击者可以用这个做什么"，不只是"这里有个异常"
✅ 脱敏截图 截图中的 token、cookie、邮箱、真实 ID 全部打码
✅ 英文报告格式 按 english_report_template.md 的格式：标题、摘要、步骤、影响、建议修复
✅ 未触碰真实用户 整个测试过程只用测试账号，没有读取或修改任何真实用户数据
