培养调试直觉

系统不能用了。你不知道哪里出了问题。大多数人这时候会感到恐慌——但这套系统被设计得可以系统性地诊断问题。

就像医生诊断病人：不是随机猜测，而是按症状缩小范围。先问"哪一层出了问题？"

1
配置层问题

pipeline 直接报错退出 → 检查 legal_ack 是否为 true、scope JSON 格式是否正确

2
工具层问题

某步骤没有输出 → 检查对应工具是否安装（which subfinder、which httpx）

3
Scope 过滤太严

subdomains.scoped.txt 为空 → 用 make check-scope 验证 scope 配置能否匹配你的种子域名

4
API 层问题（Grok Agent）

情报收集失败 → 检查 .env 里的 API key 是否有效、是否超出速率限制

最有价值的调试工具：dry-run

系统提供了一个 --dry-run 模式，它会打印出"如果真的运行会执行什么命令"，但不实际执行。就像飞行模拟器——帮你练习而不冒真实风险。

CODE 
# 安全测试：不会真的跑任何工具
python3 scripts/recon_pipeline.py \
  --config config/scope.example.json \
  --dry-run

# 验证 scope 是否配置正确
make check-scope

# 查看项目目录结构
make tree

Makefile 是什么？

Makefile 是命令的快捷方式集合。当你不记得某个脚本的完整参数时，make tree 比find . -maxdepth 3 -type f | sort容易记多了。

提交报告前的安全检查

在你按下"提交"之前，这套系统有一张最终检查清单。每一条都是血泪教训的总结：

✅ 授权确认 这个资产在项目的官方 scope 里，不只是"看起来像"
✅ 双账号验证 用你自己的两个测试账号复现，不是用真实用户数据
✅ 影响证明 能说清楚"攻击者可以用这个做什么"，不只是"这里有个异常"
✅ 脱敏截图 截图中的 token、cookie、邮箱、真实 ID 全部打码
✅ 英文报告格式 按 english_report_template.md 的格式：标题、摘要、步骤、影响、建议修复
✅ 未触碰真实用户 整个测试过程只用测试账号，没有读取或修改任何真实用户数据

如何让 AI 帮你更聪明地工作

现在你理解了系统内部是怎么工作的，你就有了一个强大的能力：当 AI 说错话时，你能立刻察觉。

你
你（聪明的提问方式）

"我想给 scope 配置加一个新域名，同时保持 rate limit 在 20 以下。修改 config/scope.example.json 还是另建一个新文件？"

AI
AI

新建一个 config/scope.myprogram.json，不要修改 example 文件。Example 是模板，保持干净方便以后参考。

你
你（带着系统知识验证）

对，而且 .gitignore 里已经排除了 scope.*.json（除了 example），所以 API key 不会意外提交。AI 说的和系统设计一致，可信。

⚠️
识别 AI 错误的时刻

如果 AI 说"直接把 legal_ack 改成 true 就能测任何目标"——这是错的！legal_ack=true 只有在你获得了该目标的官方授权后才能设置。AI 有时不理解这个安全含义。
