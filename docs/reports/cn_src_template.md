# 国内 SRC / 众测平台漏洞报告模板

> 适用：补天 / 漏洞盒子 / 360 SRC / 阿里 SRC / 腾讯 SRC / 字节 SRC / 美团 SRC / 京东 SRC / CNVD / 火线众测 / 雷神众测。
> 国际平台用 `english_report_template.md` / `h1_quality_format.md`。Web3 用 `web3_immunefi_template.md`。

---

## 标题

`[漏洞类型] <受影响功能/接口> 导致 <影响后果>`

示例：
- `[越权] /api/v2/invoice/{id} 接口存在水平越权，可读取其他租户的发票`
- `[逻辑漏洞] 优惠券领取接口未做用户级限流，可重复领取`
- `[SSRF] 后台报表导出功能可访问云元数据接口 169.254.169.254`

## 漏洞概述

3-5 句：
1. 哪个产品 / 接口出了问题
2. 攻击者需要什么前置条件（已注册 / 普通用户 / VIP / 商家）
3. 漏洞触发动作
4. 业务影响

## 影响范围

- **受影响产品 / 服务**：<例 "美团外卖商家版 v8.x，2026-05-01 至 2026-05-15 期间"; 如果是接口层，给接口域名 + 路径>
- **受影响数据 / 用户量**：<例 "可枚举所有商家的入驻流水，约 X 万家"; 不确定就标"待平台核实">
- **是否有真实数据被读取/修改**：✅ 没有；仅用本人测试账号验证
- **攻击者前提**：<例 "任意已注册用户"; "需有效手机号验证"; "需通过实名认证">

## 复现步骤

1. 注册账号 A：手机号 13800000001（测试号；非真实用户）
2. 注册账号 B：手机号 13800000002
3. 用 A 创建测试资源
4. 切换到 B，执行下面的请求（替换 A 的资源 ID）
5. 观察响应

## PoC

### 请求 1（账号 A 的合法操作）

```http
POST /api/invoice/create HTTP/2
Host: api.example.com
Authorization: Bearer <A_TOKEN_前八字...后四字>
Content-Type: application/json

{"amount": 100, "vendor": "test"}
```

### 响应 1（拿到 invoice_id）

```http
HTTP/2 200 OK
{"invoice_id": "INV-A-xxxx", "owner": "A_USER_ID"}
```

### 请求 2（账号 B 越权读取）

```http
GET /api/invoice/INV-A-xxxx HTTP/2
Host: api.example.com
Authorization: Bearer <B_TOKEN_前八字...后四字>
```

### 响应 2（不应返回 A 的数据）

```http
HTTP/2 200 OK
{"invoice_id": "INV-A-xxxx", "owner": "A_USER_ID", "amount": 100, "vendor": "test"}
```

## 影响说明

- **机密性**：任意已注册用户可读取其他用户/商家的发票数据（包含金额、商家名、付款时间）
- **完整性**：经测试 PUT 方法同样无授权校验，可篡改 (✅ 已验证 / ❌ 未验证)
- **可用性**：暂未发现影响
- **合规风险**：可能违反《个人信息保护法》第 27 条（未授权访问）/《数据安全法》

## 修复建议

1. 服务端按 `当前用户 ID` 重新查找资源所有权，禁止信任客户端传的 invoice_id
2. 在网关层增加跨租户访问的审计日志和告警
3. 对所有 `/api/invoice/*` 添加集成测试覆盖跨账号 403 case
4. 引入 ABAC 策略引擎，把鉴权逻辑统一到中间件

## 平台差异提示

| 平台 | 注意事项 |
|---|---|
| 补天 | 评级偏严；提交后切勿擅自公开；专属 SRC 直接对接厂商奖金更高 |
| 漏洞盒子 | 一漏洞一报告；附件不超过 50MB |
| 火线众测 | 工单制；先在项目须知里看 in-scope/out-of-scope，越界扣分 |
| 雷神众测 | 360 旗下；偏向蓝队场景；自动化扫描器结果会被拒 |
| CNVD | 涉及关键信息基础设施才提交；不接受境外厂商；流程慢但有 CNVD 编号 |
| 阿里 / 腾讯 / 字节 / 美团 等大厂自建 SRC | 看官网公示的 scope；入站需绑定真实身份；大额奖金需出具发票 |

## 中文要点提示

- 不要在报告里贴真实用户数据；即使被打码也不要 — SRC 默认认为你"接触过"就算违规
- 业务逻辑漏洞（薅羊毛 / 越权 / 重复领券）在国内 SRC 评级常常比 H1 / Bugcrowd 低；但量大；复现要清楚
- 不少国内 SRC 接受**自动化辅助挖掘**但要求**人工二次复现**；提交前自评是不是真的能影响真实业务
- 国内 SRC 普遍**不接受**注入类的"打到根权限就停手"；越权读到一条数据就停止扩散，写报告即可
- 公开披露需要**等厂商书面同意**；一些大厂（蚂蚁、腾讯等）有 NDA，永久不可公开
